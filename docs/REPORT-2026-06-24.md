# Strata / Encarta ETL — progress report (2026-06-24)

Status of the Encarta 2009 reverse-engineering + ETL pipeline across the two
repos: **`akc`** (`strata-works/akc`, the portable AKC/LIT decoder) and
**`quarry`** (`strata-works/quarry`, the SQLite/FTS corpus builder).

## TL;DR

The text corpus is **complete and production-ready**: 116,119 articles with
titles, full-text search, cross-references, media links, and a 409,937-asset
store — all UTF-8-clean and joinable. The decoder is fast (native LZX ~185×; AKC
~25× via PyPy, byte-identical). The renderer spec (tag vocabulary) is done. The
remaining work before a reader UI is purely product/UI; on the data side only
media transcoding (NEX-395) and minor asset reclassification are open.

## Work completed this cycle

### 1. AKC decode performance — PyPy (~25×, byte-identical)

The pure-Python AKC record decoder (`CanonicalHuffman.decode`, a bit-by-bit
Huffman walk) was the build-time bottleneck (~76 min content build; the media
phase had to be killed). Resolved by running the AKC phases under **PyPy 7.3.17**
(JIT) — no code change, no porting:

- Codec benchmark (`akc/bench_akc_decode.py`), 3,000 CONTSTD records, cold:
  **CPython 3.14 = 152.5 s vs PyPy = 5.96 s (25.6×)**, identical parity sha1.
- This is clean because the AKC path has **zero native dependencies**; the
  LZX/EIT phase still runs under CPython + the native `lzx.so`, and the two
  phases only meet through `encarta.sqlite` on disk.
- Native arm64 LZX (calibre/libmspack, committed earlier) remains ~185×,
  byte-identical.

### 2. Media build — validated end-to-end, then fully filled

- Validated through the real quarry path (`bench_media_build.py`): DATASTD,
  2,000 records, **CPython 105.9 s vs PyPy 4.62 s (22.9×)**, full-DB sha1
  identical between interpreters.
- Full media fill (`fill_media.py`, PyPy): **307,183 records, 0 failures, 634 s**
  (vs ~4.5 h on CPython — why it kept getting killed). Discovery is whitelisted
  to `DATA_FAMILIES {DATASTD, DATADLX, DATASTC, DATAKDC}`, so the broken
  `DATAF*` table-reuse variants are skipped automatically.
- Result: `media`/`media_file`/`article_media` populated; **all 514,398
  `media_file` rows resolve to a stored asset** — the NEX-392 article↔asset join
  works end-to-end.

### 3. Article titles — resolved (NEX-394 sub-task)

Article titles are **not** in the body XML (`<inlinetitle>` is empty; only ~2%
have `<headline>`). Investigated `CATALOG.STE` — extracted it and confirmed it
holds **no titles**, only GUIDs + binary index tables (dead end).

The titles were already decoded into the DB: each article's **own DATA* record**
(`media.refid == article.refid`) carries its display title. So resolution was a
**backfill, not reverse-engineering**:

- New `store.backfill_article_titles()` + `title_count()`; `build_corpus`
  backfills after the content+media passes.
- **116,116 / 116,119 titles populated** (3 stubs have no DATA title).

### 4. UTF-8 encoding bug — fixed corpus-wide

The AKC decoder emits **UTF-8**, but `media.py` and `content.py` decoded as
**windows-1252** (the "windows-1252 confirmed" note in NEX-394 was wrong —
verified against raw DATASTD bytes: U+2019 `'` is `e2 80 99`, not `0x92`). This
produced mojibake (`A Bugâ€™s Life`) in `media.title` and the FTS body text. Raw
`article.xml` was already correct UTF-8 — only derived strings were damaged.

- Fixed both decoders (TDD: `test_decodes_utf8_title_with_smartquote`,
  `test_fts_text_decodes_utf8`).
- Corrected rebuild (content + media + backfill, PyPy, **698 s**): **0 mojibake
  titles**, `quick_check ok`, `461510896 → A Bug's Life`.

### 5. Tag vocabulary survey — NEX-393 (Done)

`docs/VOCABULARY.md`: **32 elements** surveyed over all 116,119 bodies —
frequency, attributes, example, nesting, class (root/block/inline/xref/media/
meta). Highlights:

- `pkey` = paragraph unit; `section` (type 4–7) nests; `sec/seca/secb/secc` =
  outline enumerators.
- `xref`: type 8 = internal article link (RefID→article), type 9 = external URL;
  `paraID` = deep paragraph link.
- `inlinetitle` is an empty placeholder → renderer fills it with `article.title`.
- Media: **no `msencdata:` URLs in bodies** — article media via the
  `article_media` join; inline images via `inlinebmp id="<NAME>.DIB"`.
- **XSL is embedded in `MSENCXML.DLL`** (no loose `.xsl` in the tree).
  **Recommendation: build our own renderer from the semantic XML** (the 32-tag
  vocabulary maps cleanly to widgets); don't port the Windows-DLL XSL.

### 6. Contentless FTS — DB shrunk ~43%

`article_fts` stored a redundant second copy of the body text (the 392 MB
`article_fts_content` shadow). Switched to a **contentless FTS5** table
(`content='', contentless_delete=1` — the latter preserves the idempotent
delete-then-insert re-ingest). Search is unaffected (returns rowids ranked by
bm25; no stored text needed).

- TDD: `test_fts_is_contentless_keeps_no_body_copy`.
- Migration (`migrate_fts_contentless.py`): rebuilt the index from stored
  `article.xml` (no AKC re-decode, 29.8 s) + VACUUM.
- **DB: 1,268.8 MB → 717.8 MB (saved 551 MB).** Verified: `quick_check ok`,
  fts_count 116,119, search returns 493 hits for "Ottoman Empire".

#   akc CPython venv: .venv (native LZX)   |   PyPy venv: .venv-pypy (fast AKC)
PYPY=../akc/.venv-pypy/bin/python

$PYPY fill_media.py             # DATA* -> media/media_file/article_media (PyPy)
$PYPY rebuild_text.py           # content + media + title backfill (UTF-8)
$PYPY migrate_fts_contentless.py# rebuild article_fts contentless + VACUUM
.venv/bin/python -m unittest discover -s tests   # 30 tests
